关于Linux连接跟踪模块

一、背景概述

nf_conntrack模块在kernel 2.6.15（2006-01-03发布）被引入，支持ipv4和ipv6，取代只支持ipv4的ip_connktrack，用于跟踪连接的状态，供其他模块使用。

最常见的使用场景是 iptables 的 nat 和 state 模块：

  • nat 根据转发规则修改IP包的源/目标地址，靠nf_conntrack的记录才能让返回的包能路由到发请求的机器。
  • state 直接用 nf_conntrack 记录的连接状态（NEW/ESTABLISHED/RELATED/INVALID）来匹配防火墙过滤规则。

调整server端配置要重启80端口的服务，为使重启过程中请求无损但又不想从负载均衡member中剔除，于是想采用iptables禁用和80端口新建连接的方式阻止负载均衡的健康监测，从而达到摘除的目的。
但是在操作过程中出现了iptables连接跟踪模块哈希表溢出的现象，原因是因为在拦截流量完成后通过iptables -F清空规则的时候默认开启了防火墙记录所有连接状态的功能，因流量过大导致丢包的发生。

二、问题出现

  • 执行iptables命令：
iptables -A INPUT -m state --state NEW -p tcp --dport 80 -j DROP
  • 流量逐渐下降，此时日志中没有出现错误码
  • 调整配置
  • 重启服务
  • 执行iptables命令清空规则
iptables -F
  • message中出现大量报错：
Jul 11 17:38:36 73-14-215-tc-core kernel: nf_conntrack: table full, dropping packet

nf_conntrack用1个哈希表记录已建立的连接，包括其他机器到本机、本机到其他机器、本机到本机（例如 ping 127.0.0.1 也会被跟踪）。
如果连接进来比释放的快，把哈希表塞满了，新连接的数据包会被丢掉，此时netfilter变成了一个黑洞，导致拒绝服务。 这发生在3层（网络层），应用程序毫无办法。

三、解决方法

关于优化iptables连接跟踪模块分为两种情况：

1、调整内核参数

通过参数优化的的手段为两种：

  1. 放大链接跟踪hash表的容量
  2. 减小每条filter链接的在hash表中的超时时间

参考：
https://www.kernel.org/doc/Documentation/networking/nf_conntrack-sysctl.txt

参数说明

  • net.netfilter.nf_conntrack_buckets

跟踪的连接用哈希表存储，每个桶（bucket）里都是1个链表，默认长度为4。即平均每个链表里有4个filter连接。
netfilter的哈希表存储在内核空间，这部分内存不能swap
操作系统为了兼容32位，默认值往往比较保守：在32位Linux下，内核空间的虚拟地址空间最多 1G，通常能用的只有前 896M

增加 bucket 数量后，利用空间换时间的方法，将少量长链表换成多个短链表，单个链表检索速度会大幅度下降

默认值：
4G内存以下的默认值(最大占用1G)：16384
4G内存以上的为： 4 * 1024^3 / 16384 = 65536

nf_conntrack_buckets - INTEGER
Size of hash table. If not specified as parameter during module
loading, the default size is calculated by dividing total memory
by 16384 to determine the number of buckets but the hash table will
never have fewer than 32 and limited to 16384 buckets. For systems
with more than 4GB of memory it will be 65536 buckets.
This sysctl is only writeable in the initial net namespace.

  • net.netfilter.nf_conntrack_max

默认值：

net.netfilter.nf_conntrack_buckets * 4

同上，如果4G内存以下的默认值(最大占用1G)：16384 * 4 = 65536

nf_conntrack_max - INTEGER
Size of connection tracking table. Default value is
nf_conntrack_buckets value * 4.

内存占用情况：
一条跟踪记录占用内存16384bytes，最大为

CONNTRACK_MAX = RAMSIZE (in bytes) / 16384

一般情况下是65536(最多占用1G内存)：

1 * 1024 * 1024 * 1024 / 16384 = 65536
  • net.netfilter.nf_conntrack_tcp_timeout_established

默认值： 5天

理论上不用这么长，不小于 net.ipv4.tcp_keepalive_time 就行了

nf_conntrack_tcp_timeout_established - INTEGER (seconds)
default 432000 (5 days)

根据实际情况最终参数为（假设最大允许占用4G内存，TCP连接半小时超时）：

$ vi /etc/sysctl.conf
net.nf_conntrack_max = 262144
net.netfilter.nf_conntrack_buckets = 65536
net.netfilter.nf_conntrack_tcp_timeout_established = 1800

状态查看

查看当前跟踪连接数

cat /proc/net/nf_conntrack | wc -l

统计里面的TCP连接的各状态和条数

cat /proc/net/nf_conntrack | awk '/^.*tcp.*$/ {count[$6]++} END {for(state in count) print state, count[state]}'

记录数最多的10个ip

cat /proc/net/nf_conntrack | awk '{print $7}' | cut -d "=" -f 2 | sort | uniq -c | sort -nr | head -n 10

优缺点分析

  • 优点：不影响防火墙的工作

  • 缺点：设置过大会占用一部分内存，同时因conntrack的hash表过大导致系统的查找效率降低引发通信效率低的问题

2、禁用nf_conntrack模块

如果不使用iptables中根据状态判断的规则(–state等)，可直接删除相关模块

手动删除这些模块下面这些内核模块：

cat /etc/modprobe.d/blacklist.conf  |awk '{print "modprobe -r "$2}' |bash

相当于

modprobe -r nf_conntrack
modprobe -r nf_conntrack_ipv6
modprobe -r xt_conntrack
modprobe -r nf_conntrack_ftp
modprobe -r xt_state
modprobe -r iptable_nat
modprobe -r ipt_REDIRECT
modprobe -r nf_nat
modprobe -r nf_conntrack_ipv4

直接把iptables相关的模块加入黑名单，避免下次重启加载：

$ cat /etc/modprobe.d/blacklist.conf
blacklist nf_conntrack
blacklist nf_conntrack_ipv6
blacklist xt_conntrack
blacklist nf_conntrack_ftp
blacklist xt_state
blacklist iptable_nat
blacklist ipt_REDIRECT
blacklist nf_nat
blacklist nf_conntrack_ipv4

如果上述相关模块在一开始就被禁用，则在使用iptables命令的时候会报如下错误，此时使用modprobe MOD_NAME 来增加相应的模块即可

$ iptables -A INPUT -m state --state NEW -p tcp --dport 80 -j DROP
iptables: No chain/target/match by that name.
