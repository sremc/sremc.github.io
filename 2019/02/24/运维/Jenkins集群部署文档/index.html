Jenkins 集群部署文档

1. 概述

本文档是基于Jenkins平台，对Java工程进行docker化部署，实现从Java源代码到Docker镜像的整个构建过程。

1.1. 环境说明

名称 版本
Jenkins 2.93
Docker 1.12.6
Maven 3.5.2
JDK 1.8.0_102

1.2. Jenkins工作流程图

2. 部署

2.1. Jenkins镜像构建

为方便在生产环境的快速部署和故障迁移等工作，我们Jenkins自身也选用docker化部署。

2.1.1. 获取基础镜像和JDK

#获取CentOS镜像
docker pull docker.io/centos:latest

#下载JDK-1.8
http://www.oracle.com/technetwork/java/javase/downloads/jdk8-downloads-2133151.html

cd /path/to/jenkins
java -jar --httpPort=8081 jenkins.war

2.1.2. 构建Jenkins镜像

构建Jenkins镜像准备以下三个文件，分别是：

  • 容器初始化脚本-container_init.sh

为方便Jenkins的快速迁移和，提高部署的灵活性，我们在初始化脚本中要求强制指定端口号，端口号通过环境变量设置和获取

$ cat container_init.sh
#!/bin/bash
source /etc/profile
if [[ -z "$JENKINS_PORT" ]];then
    echo "JENKINS_PORT is null"
    exit
fi
java -jar /usr/local/jenkins/jenkins.war -ajp13Port=-1 --httpPort=$JENKINS_PORT
  • Dockerfile
$ touch Dockerfile
$ cat Dockerfile
FROM google/base-image-jdk
MAINTAINER smc2@qq.com
RUN unlink /etc/localtime
RUN ln -s  /usr/share/zoneinfo/Asia/Shanghai /etc/localtime
RUN mkdir -vp /usr/local/jenkins
COPY jenkins.war /usr/local/jenkins/
COPY container_init.sh /
RUN chmod -v 755 /container_init.sh
ENTRYPOINT /container_init.sh
  • Jenkins程序包
wget -c http://updates.jenkins-ci.org/download/war/2.73/jenkins.war

2.1.3. 执行构建命令

docker build -t ${harbor-server}/${project-name}/jenkins-image:1.0 .

2.1.4. 把镜像上传到私有仓库

docker push  ${harbor-server}/${project-name}/jenkins-image:1.0

2.2. 启动Jenkins

登录Jenkins服务器启动Jenkins服务(假设申请的端口为18040)

#把Jenkins的默认的JENKINS_HOME挂载到宿主机，便于调整配置和备份
mkdir -vp /data/jenkins_home

#通过环境变量指定Jenkins运行的端口
docker run -d  -e "JENKINS_PORT=18040" -v /data/jenkins_home:/root/.jenkins -v /etc/resolv.conf:/etc/resolv.conf -v /etc/hosts:/etc/hosts  --name jenkins --net=host ${harbor-server}/${project-name}/jenkins-image:1.0
/container_init.sh
  • 通过浏览器访问如下地址，进行Jenkins的初始化和管理员密码的设定
http://server-ip:18040

去服务器上拷贝红字部分的文件中的初始密码，填入，点击继续：

2.3. 配置Jenkins PipeLine

Jenkins 2.0的到来，pipline进入了视野，jenkins2.0的核心特性. 也是最适合持续交付的feature。
简单的来说，就是把Jenkins1.0版本中，Project中的相关配置信息，如MVN/Git的配置，Parameter的配置等都变成Code，即Pipeline as Code。

更多详细解释参见官方文档：
https://jenkins.io/doc/book/pipeline

2.3.1. 插件配置

路径： 系统管理 -> 插件管理 -> 可选插件
搜索并安装如下插件：
Pileline plugin

docker plugin

maven plugin

SSH Slaves plugin

2.3.2. 任务配置

路径：首页 -> 新建

填入相关名字，选择Pipeline ，点击OK， 在Pipeline处的Definition处选择Pipeline script，填入如下代码：

每个stage即为一个阶段或步骤，下面代码中主要分为三个阶段，分别为 clone and compile， build image， deploy， 最后一个阶段要结合具体环境选择，比如在开发或者测试环境构建完成后直接部署到开发或者测试环境，但是在生产环境中则不可这么做。

具体步骤的细节解释在代码中的注释中（注释用//开头）

node {
    //定义构建过程中的需要用到的变量

    //Harbor仓库相关的配置
    def harborServer = "10.46.235.125:1804"
    def harborUsername = "admin"
    def harborPassword = "harborPassword"

    //自定义组合版本号
    def trunkVersion = "0"
    def newImageTag = "migu/sociality-core:${trunkVersion}.${env.BUILD_ID}"

    //也可以使用项目中的pom.xml配置文件中的version标签内的版本号

    //服务检测信息，对于构建完成的镜像进行初步测算，如果在18004端口出现下面的`No message available`则任务镜像构建成功
    def servicePort = "18004"
    def serviceCheckKeyWord = "No message available"

    //有Jenkins是运行在docker内部，所以构建镜像的任务需要在远程docker执行，即Jenkins容器所在的宿主机的docker，如下所示，docker进程必须监听在2376端口
    def dockerDaemonHost = "127.0.0.1:2376"

    //此步骤是把源代码编译成jar包
    stage('clone and compile'){
        //定义maven插件的命令
        def mvnHome = tool 'M3'
        env.PATH = "${mvnHome}/bin:${env.PATH}"

        //从git服务器拉取代码
        echo "pull source code"
        sh '''
        [[ -d /tmp/git-tmp ]] && rm -rf /tmp/git-tmp
        mkdir -vp /tmp/git-tmp
        git clone http://usernaem:password@10.46.235.11/server/mgdm/sociality.git /tmp/git-tmp/
        /usr/bin/rsync -az /tmp/git-tmp/ ./
        '''

        //使用maven编译代码
        echo "compile with maven"
        sh "mvn -v"
        sh "mvn clean package  -Dprofile.id=test"
    }

    //此步骤是把jar包打包成docker镜像
    //Jenkins所在的宿主机要有一个基础镜像，包含除了Java工程之外任何所需的环境，比如JDK，环境变量等
    //Dockerfile的作用就是把本地编译好的jar包工程拷贝到基础镜像中，再打包成新镜像
    stage('build image') {
                //把Dockerfile写入本地文件
                writeFile (
                    file: 'Dockerfile',
                    text:
'''
FROM 10.46.235.125:1804/migu/base-image-jdk:1.0
MAINTAINER admin@xxx.com
ENV LANG en_US.UTF-8
ENV JAVA_HOME /usr/java/jdk1.8.0_151
ENV PATH $JAVA_HOME/bin:$PATH
RUN mkdir -vp /usr/local/run
COPY sociality-core/target/sociality-core-*.jar /usr/local/run/sociality-core.jar
EXPOSE 18004
ENTRYPOINT java -jar /usr/local/run/sociality-core.jar
''')

        //主要这里要用指令`docker.withServer`，因为我们的Jenkins是安装在docker容器中的
        docker.withServer("tcp://${dockerDaemonHost}") {
            echo "Build Docker Image"
            def image = docker.build "${newImageTag}"

            //测试镜像的原理是在宿主机中启用一个临时的容器用于测试新镜像的正确性，测试结束之后临时容器随即被删除，如果测试失败，脚本返回的状态码非零，任务则会失败，不会继续往下执行push操作
            //此部分内容可参见： https://issues.jenkins-ci.org/browse/JENKINS-41316
            echo "Test Docker Image"
            docker.image("${newImageTag}").withRun(" -v /etc/resolv.conf:/etc/resolv.conf -v /etc/hosts:/etc/hosts --name migu_for_test_tmp --net=host --expose=${servicePort} ") {

                echo "start test new image"
                sh '''
                    nohup ENTRYPOINT java -jar /usr/local/run/sociality-core.jar &
                '''
                //服务测试脚本内容写到本地
                writeFile (
                    file: 'check_service.sh',
                    text:
'''#!/bin/bash
sport=$1
kword=$2
echo "$(date +%F-%T) start checking service"
service_ok=false
for i in $(seq 60);do
    echo "$(date +%F-%T) checking service for $i times"
    if curl -s -m 5 localhost:$sport/ | grep -i "$kword";then
        echo "$(date +%F-%T) service check success"
        service_ok=true
        break
    else
        sleep 5
    fi
done

if $service_ok ; then
    exit 0
else
    echo "$(date +%F-%T) check service failed"
    exit 2
fi
''')
                sh "/bin/bash check_service.sh ${servicePort} \"${serviceCheckKeyWord}\""
            }

            //上述对新镜像的测试通过后，开始执行镜像的push操作，把新生成的镜像push到私有仓库
            echo "Publish Docker Image"
            docker.withRegistry("http://${harborServer}") {
                sh "docker login --username ${harborUsername} --password ${harborPassword} ${harborServer}"
                image.push()
            }
        }
    }

    //此步骤是部署环节，根据环境中使用的容器管理工具，可调用部署容器的API或者直接触发更新脚本新镜像的部署
    //要结合具体环境选择，比如在开发或者测试环境构建完成后直接部署到开发或者测试环境，但是在生产环境中则不可这么做。
    stage('deploy'){
        echo "deploy ......"
    }
}

点击OK，保存配置

3. Jenkins主从配置

jenkins支持主从模式，这将会把构建任务分发到多个从节点去执行，这样就可以支撑起多个项目的大量构建任务，同时，你可以提供多种环境（如：开发环境、生产环境）来对同一个项目进行测试和构建。

3.1. 准备工作

  1. 确保安装了如下插件

    • SSH Slaves plugin
    • Multi slave config plugin

  2. 确保通过SSH下发子任务的用户拥有对远程工作目录的读写权限

  3. 确保slave节点上的git和maven可执行文件已经安装，为避免过多麻烦尽量安装在和master一致的位置
